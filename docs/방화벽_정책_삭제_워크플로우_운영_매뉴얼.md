## 1. 업무 개요

### 1.1 목적

방화벽에 등록된 수천~수만 건의 정책 중 **삭제 대상**을 체계적으로 선별하여 정책 관리자·신청부서에 공지하는 작업이다.  
삭제 대상은 크게 세 가지 유형이다.

| 삭제 유형 | 판단 기준 |
|---|---|
| **기간만료 + 사용** | 신청 기간이 만료됐으나 여전히 트래픽이 발생 중인 정책 |
| **기간만료 + 미사용** | 신청 기간이 만료됐고 트래픽도 없는 정책 |
| **장기미사용** | 기간은 유효하나 90일 이상 트래픽이 없는 정책 |
| **이력없는 미사용** | 신청 이력 자체가 없고 트래픽도 없는 정책 |

### 1.2 지원 벤더

- **Palo Alto Networks** (PA 시리즈, HA 구성 포함)
- **SECUI MF2**
- **SECUI NGF**

## 3. 준비 사항 및 필요 파일

### 3.1 필수 외부 파일

| 파일명 | 획득 방법 | 사용 단계 |
|---|---|---|
| **GSAMS 신청정보 Excel** | GSAMS 포털 다운로드 (전체 기간, 전체 부서) | Task 7 |
| **MIS ID 매핑 CSV** | IT 자산관리 시스템 내보내기 (`ruleset_id`, `mis_id` 컬럼 필요) | Task 5 |
| **HA Secondary 히트카운트** | Palo Alto HA 구성 시 Secondary 장비에서 별도 추출 | Task 1 (선택) |
| **중복예외 YAML** | 이전 분기 Task 15 산출물 또는 수동 작성 | Task 17 (선택) |

### 3.2 MIS ID CSV 형식

```
ruleset_id,mis_id
P2024-001234,MIS-5678
P2024-001235,MIS-5679
```

### 3.3 작업 환경

- Excel 2016 이상 (`.xlsx` 형식)
- Python 3.10 이상 (자동화 도구 사용 시)
- FAT 시스템 접속 권한

## 4. Phase 0 — 데이터 추출

### Task 0. FAT DB 추출

**목적**: 방화벽 현재 정책 목록과 사용이력(히트카운트)을 한 파일에 담은 기준 데이터를 생성한다.

**처리 절차**:
1. FAT 시스템 → 해당 장비 선택 → "DB 추출" 실행
2. 생성 파일: `YYYY-MM-DD_장비IP_policy.xlsx`

**출력 파일 시트 구성**:

| 시트명 | 내용 |
|---|---|
| `policy` | 정책 목록 (Rule Name, Action, Source, Destination 등) |
| `usage` | 히트카운트 (Rule Name, Hit Count, Last Hit Date, Unused Days) |
| `address` | 주소 객체 |
| `address-group` | 주소 그룹 객체 |
| `service` | 서비스 객체 |
| `service-group` | 서비스 그룹 객체 |

**`policy` 시트 핵심 컬럼**:

| 컬럼명 | 설명 | 예시 |
|---|---|---|
| `Vsys` | 가상 시스템 | `vsys1` |
| `Seq` | 정책 순번 (위에서부터 1, 2, 3…) | `1` |
| `Rule Name` | 정책 이름 (Description에 신청 정보 포함) | `P2024001234-20241231-hong_gildong` |
| `Enable` | 활성화 여부 | `Y` / `N` |
| `Action` | 허용/차단 | `allow` / `deny` |
| `Source` | 출발지 IP/객체 | `10.0.0.1,10.0.0.2` |
| `Destination` | 목적지 IP/객체 | `192.168.1.0/24` |
| `Service` | 서비스/포트 | `tcp_8080,tcp_443` |
| `Description` | 신청 정보 원문 (파싱 대상) | `P0001234;20240101~20241231;hong;MIS-5678` |

---

### Task 1. HA 히트카운트 병합 (Palo Alto HA 장비 한정, 선택)

**목적**: Active-Passive HA 구성인 경우 Primary와 Secondary 히트카운트를 합산하여 더 정확한 사용이력을 산출한다.

**처리 절차**:
1. Secondary 장비에서 히트카운트 Excel 별도 추출
2. Primary 히트카운트 파일 + Secondary 히트카운트 파일을 입력으로 실행

**병합 규칙**:

| 컬럼 | 병합 방법 |
|---|---|
| `Hit Count` | Primary + Secondary 합산 |
| `Last Hit Date` | Primary와 Secondary 중 **최신 날짜** 선택 |
| `Unused Days` | Primary와 Secondary 중 **더 적은 값** 선택 (보수적 기준) |
| `미사용여부` | Unused Days > 90일이면 `미사용`, 이하면 `사용` |

**출력**: `Merged_Primary파일명.xlsx`

> HA 구성이 아닌 단일 장비는 Task 1 생략, DB 추출 파일의 `usage` 시트를 직접 사용한다.

## 5. Phase 1 — 정책 파일 전처리

### Task 2. 신청정보 파싱

**목적**: 정책의 `Rule Name` 또는 `Description` 필드에서 신청 정보(신청번호, 신청자, 기간 등)를 추출하여 별도 컬럼으로 분리한다.

**입력**: Task 0 DB 추출 파일 (`policy` 시트)

**파싱 대상 형식**:

| 형식 | 구분 코드 | 예시 |
|---|---|---|
| GSAMS 3.0 (현행) | `P`=GROUP, `F`=GENERAL, `S`=SERVER, `M`=PAM | `P0001234;...` |
| GSAMS 1.0 (구형) | 룰이름 또는 Description 내 특수 패턴 | `OLD형식` |

**추가되는 컬럼**:

| 컬럼명 | 설명 | 비고 |
|---|---|---|
| `Request Type` | 신청 유형 | `GROUP` / `GENERAL` / `SERVER` / `PAM` / `OLD` / `Unknown` |
| `Request ID` | 신청번호 | `P2024-001234` |
| `Ruleset ID` | 룰셋 ID | `P0001234` |
| `MIS ID` | MIS 자산 ID (이 단계에서는 비어있을 수 있음) | `MIS-5678` |
| `Request User` | 신청자 사번/ID | `hong_gildong` |
| `Start Date` | 신청 시작일 | `2024-01-01` |
| `End Date` | 신청 종료일 | `2024-12-31` |

**출력**: `정책파일_v1.xlsx`

---

### Task 3. 중복정책 분석 (FAT 자동)

**목적**: 동일한 출발지·목적지·서비스 조합을 가진 중복 정책을 탐지하고 그룹 번호(`No`)를 부여한다.

**처리**: FAT 시스템 내부 분석 엔진이 자동 처리 (별도 조작 불필요)

**출력 컬럼**:

| 컬럼명 | 설명 |
|---|---|
| `No` | 중복 그룹 번호 (같은 No끼리 중복 관계) |
| `Type` | `Upper` (상단, 실제 적용되는 정책) / `Lower` (하단, 가려진 정책) |
| 기존 정책 컬럼 | 그대로 유지 |

---

### Task 4. 중복결과 신청정보 파싱

**목적**: Task 3 중복분석 결과 파일에도 신청정보를 파싱하여 추가한다.  
**입력**: Task 3 출력 파일  
**처리**: Task 2와 동일한 파싱 로직 적용  
**출력**: `중복결과_v1.xlsx`

### Task 5. MIS ID 매핑

**목적**: IT 자산관리 시스템의 MIS ID를 `Ruleset ID` 기준으로 정책 파일에 채워넣는다.

**입력**:
- Task 2 출력 (`정책_v1.xlsx`)
- MIS ID 매핑 CSV

**매핑 규칙**:
- `Ruleset ID`를 키로 CSV에서 `mis_id`를 조회
- 정책 파일의 `MIS ID` 컬럼이 비어있는 행에만 채움 (기존 값 보존)
- CSV에 중복 `ruleset_id`가 있으면 첫 번째 항목만 사용

**출력**: `정책_v2.xlsx`

---

### Task 6. 신청번호 추출

**목적**: 이후 GSAMS 조회에 사용할 신청번호 목록을 추출한다.

**입력**: Task 5 출력 (`정책_v2.xlsx`)

**처리**: `Request Type`별로 시트를 나눠 `Request ID` 목록을 저장

**출력**: `request_id_정책파일명.xlsx`

| 시트명 | 내용 |
|---|---|
| `GROUP` | P로 시작하는 그룹 신청번호 목록 |
| `GENERAL` | F로 시작하는 일반 신청번호 목록 |
| `SERVER` | S로 시작하는 서버 신청번호 목록 |
| `PAM` | M으로 시작하는 PAM 신청번호 목록 |

## 6. Phase 2 — 신청정보 매핑

### Task 7. GSAMS 신청정보 취합

**목적**: GSAMS 포털에서 다운로드한 신청 이력 파일을 표준 포맷으로 변환한다.

**입력**: GSAMS 원본 Excel (멀티시트 가능)

**처리**:
- 각 시트를 순회하여 컬럼명을 내부 표준 컬럼명으로 매핑
- 이메일 주소 자동 생성: `WRITE_PERSON_ID@도메인`, `APPROVAL_PERSON_ID@도메인`
- `REQUEST_START_DATE`, `REQUEST_END_DATE`: YYYYMMDD → YYYY-MM-DD 변환
- 전체 시트를 하나로 합치고 `REQUEST_END_DATE` 내림차순 정렬

**출력**: `Conv_GSAMS파일명.xlsx`

**핵심 컬럼**:

| 컬럼명 | 설명 |
|---|---|
| `REQUEST_ID` | 신청번호 |
| `TITLE` | 신청 제목 (자동연장 판별에 사용) |
| `REQUEST_START_DATE` | 신청 시작일 |
| `REQUEST_END_DATE` | 신청 종료일 |
| `REQUEST_STATUS` | `99` = 자동연장 신청 |
| `WRITE_PERSON_ID` | 작성자 사번 |
| `REQUESTER_ID` | 신청자 사번 |
| `REQUESTER_EMAIL` | 신청자 이메일 |
| `MIS_ID` | 자산 ID |

---

### Task 8. 신청정보 정책파일 매핑

**목적**: GSAMS 취합 파일의 상세 신청 정보를 정책 파일의 각 정책에 매핑한다.

**입력**:
- Task 5 출력 (`정책_v2.xlsx`)
- Task 7 출력 (`Conv_*.xlsx`)

**매핑 우선순위** (GROUP 유형):

| 순위 | 매핑 기준 |
|---|---|
| 1순위 | `REQUEST_ID` 일치 AND `MIS_ID` 일치 |
| 2순위 | `REQUEST_ID` 일치 AND `REQUEST_END_DATE` 일치 AND `WRITE_PERSON_ID` 일치 |
| 3순위 | `REQUEST_ID` 일치 AND `REQUEST_END_DATE` 일치 AND `REQUESTER_ID` 일치 |

**매핑 우선순위** (GENERAL/SERVER/PAM/OLD 유형):
- `REQUEST_ID` 일치만으로 매핑

**추가되는 컬럼**:

| 컬럼명 | 설명 |
|---|---|
| `REQUEST_ID` | 매핑된 정식 신청번호 |
| `REQUEST_START_DATE` | 매핑된 실제 시작일 |
| `REQUEST_END_DATE` | 매핑된 실제 종료일 |
| `REQUESTER_ID` | 신청자 사번 |
| `REQUESTER_EMAIL` | 신청자 이메일 |
| `REQUEST_STATUS` | `99`이면 자동연장 |

**매핑 실패 시**: Task 2에서 파싱된 `Request ID`, `Start Date`, `End Date`, `Request User` 값을 그대로 사용하고 이메일은 `사번@samsung.com` 형식으로 자동 생성.

**출력**: `정책_v3.xlsx`

---

### Task 9. 자동연장 날짜 업데이트

**목적**: 자동연장이 반복된 정책의 `REQUEST_END_DATE`를 최신 연장일로 갱신한다.

**입력**:
- Task 8 출력 (`정책_v3.xlsx`)
- Task 7 출력 (`Conv_*.xlsx`)

**자동연장 판별 로직**:
1. GSAMS 파일 내에서 **앞 건의 종료일 = 뒤 건의 시작일** 조건으로 self-merge
2. 제목(`TITLE`)에서 `[TAG]` 형식 괄호 머리말 제거 후 제목 동일 여부 비교
3. 작성자(`WRITE_PERSON_ID`)가 동일한지 확인
4. 위 세 조건을 모두 만족하면 연장 체인으로 판정

**날짜 업데이트 규칙**:
- 연장 체인에서 **가장 최신 종료일**을 해당 정책의 `REQUEST_END_DATE`에 반영
- 기존 날짜보다 더 미래인 경우에만 업데이트 (소급 적용 방지)
- 자동연장 체인이 없으면 파일 버전만 올려서 다음 태스크로 전달

**출력**: `정책_v4.xlsx`

## 7. Phase 3 — 예외 분류 및 사용이력 반영

### Task 10/11. 예외처리

**목적**: 정책마다 `예외` 컬럼을 부여하여 삭제 대상에서 제외할 정책을 표시한다.

**입력**: Task 9 출력 (`정책_v4.xlsx`)

**예외 분류 기준** (우선순위 순, 뒤 항목이 앞 항목을 덮어씀):

| 예외 값 | 분류 기준 |
|---|---|
| `예외신청정책` | 시스템 설정(`request_ids`)에 등록된 신청번호와 일치 |
| `예외정책` | 시스템 설정(`policy_rules`)에 등록된 정책명과 일치 |
| `신규정책` | Rule Name에서 추출한 날짜가 최근 90일 이내 (Palo Alto) / 신청 시작일이 최근 90일 이내 (SECUI) |
| `자동연장정책` | `REQUEST_STATUS == 99` |
| `인프라정책` | 기준 정책(`deny_standard_rule_name`)보다 **위**에 위치한 정책 (Seq가 더 작음) |
| `XX정책` | 설정된 특수 접두사로 시작하는 정책 (예: `Infra_`, `BASE_`) |
| `비활성화정책` | `Enable == 'N'` |
| `기준정책` | 이름이 `_Rule`로 끝나고 비활성화된 정책 |
| `차단정책` | `Action == 'deny'` |

> **주의**: 우선순위가 낮은 항목이 높은 항목을 덮어씀. 예를 들어 `차단정책`이 `신규정책`보다 나중에 처리되므로, 차단 정책이면서 신규인 경우 `차단정책`으로 최종 분류됨.

**추가/변환되는 컬럼**:

| 컬럼명 | 설명 |
|---|---|
| `예외` | 위 표의 예외 값, 해당 없으면 빈 문자열 |
| `만료여부` | `REQUEST_END_DATE >= 오늘`이면 `미만료`, 아니면 `만료` |
| `신청이력` | `Request Type` 컬럼을 이름 변경 (GROUP/GENERAL/SERVER/PAM/OLD/Unknown) |
| `미사용여부` | 이 단계에서는 빈 문자열로 초기화 |

**제거되는 컬럼**: `Request ID`, `Ruleset ID`, `MIS ID`, `Request User`, `Start Date`, `End Date`

**출력**: `정책_vf.xlsx` (final 플래그 — 이후 버전 번호가 붙음)

### Task 12. 사용이력 반영

**목적**: 히트카운트 데이터를 정책 파일의 `미사용여부` 컬럼에 채운다.

**입력**:
- Task 10/11 출력 (`정책_vf.xlsx`)
- 히트카운트 파일 (우선순위: 수동 업로드 > Task 1 병합 결과 > Task 0 DB 추출)

**히트카운트 파일 형식** (필요한 컬럼):

| 컬럼명 | 설명 |
|---|---|
| `Rule Name` | 정책 이름 (매핑 키) |
| `미사용여부` | 직접 제공 시 (`사용`/`미사용`) |
| `Unused Days` | 미사용 일수 (간접 제공 시, `미사용여부` 없을 때 계산용) |

**미사용 판별 기준**:
- 히트카운트 파일에 `미사용여부` 컬럼이 있으면 그 값을 직접 사용
- `Unused Days`만 있으면: **90일 초과 → `미사용`**, 이하 → `사용`
- DB 추출 파일(`usage` 시트)에 `미사용여부`가 없으면 `Unused Days`로 계산

**출력**: `정책_vf_v1.xlsx`

### Task 13. 하단 최신정책 검증

**목적**: 동일 신청번호(`REQUEST_ID`) 그룹 내에서 **가장 최신 신청일의 정책이 하단(Seq가 큰 쪽)에 위치**한 경우를 탐지하고 표시한다.

**입력**: Task 12 출력 (`정책_vf_v1.xlsx`)

**탐지 로직**:
1. `REQUEST_ID`별로 그룹화
2. 그룹 내 `REQUEST_START_DATE` 최댓값 탐색 → 해당 정책들의 Seq를 `latest_seq` 로 저장
3. 그룹 내 가장 작은 Seq를 `lowest_seq` 로 추출
4. `lowest_seq`가 `latest_seq` 목록에 없으면 → 최신 정책이 하단에 있는 상황 → 해당 `latest_seq` 정책에 `미사용여부 = 하단최신정책` 표시

**실무 의미**: 동일 신청번호의 이전 정책(갱신 전)이 상단에, 최신 정책(갱신 후)이 하단에 있으면, 실제 트래픽은 상단 정책에 의해 처리되므로 하단의 최신 정책이 "사용 중"처럼 보여도 실제로는 가려진 상태임.

**출력**: `정책_vf_v2.xlsx` (멀티시트)

| 시트명 | 내용 |
|---|---|
| `usage` | 메인 정책 데이터 (미사용여부 업데이트 포함) |
| `검증` | 하단최신정책 탐지 결과 (REQUEST_ID, lowest_seq, latest_seq 목록) |

## 8. Phase 4 — 중복정책 분류

### Task 14. 중복정책 분류

**목적**: Task 3에서 탐지된 중복 그룹을 분류하여 삭제 대상과 유지 대상을 구분하고 공지용/삭제용 파일을 생성한다.

**입력**:
- Task 4 출력 — 중복분석 결과 + 신청정보 파싱 파일 (`중복_v1.xlsx`)
- Task 13 출력 — 정책 파일 (`정책_vf_v2.xlsx`)

**분류 컬럼 계산**:

| 컬럼명 | 계산 방법 |
|---|---|
| `자동연장` | `Request ID`가 자동연장 신청번호 목록에 있으면 `True` |
| `늦은종료일` | 동일 `No` 그룹 내에서 `End Date`가 가장 늦고 첫 번째인 행 = `True` |
| `신청자검증` | 동일 `No` 그룹 내 `Request User`가 모두 동일하면 `True` |
| `날짜검증` | `Type == 'Upper'` AND `늦은종료일 == True`인 `No`에 속하면 `True` |
| `작업구분` | `늦은종료일 == True` → `유지`, `False` → `삭제` |
| `공지여부` | `신청자검증 == False` (신청자가 다른 경우) → `True` (공지 필요) |
| `미사용예외` | `날짜검증 == False` AND `늦은종료일 == True` → `True` |
| `Requester Email` | 정책 파일의 `REQUESTER_EMAIL`에서 매핑 |

**예외 그룹 제거 규칙** (분류 후 최종 결과에서 제외):

| 제거 조건 | 이유 |
|---|---|
| 자동연장 GROUP이면서 동일 No 내 서로 다른 REQUEST_ID가 2개 이상 | 연장 체인 구조, 함부로 삭제 불가 |
| 자동연장이면서 GROUP 아닌 유형이 섞여있고 삭제 대상이 있는 그룹 | 자동연장 정책 보호 |
| `No` 그룹 내 모든 정책이 `작업구분 == 삭제`인 경우 | `유지` 대상이 없으면 의미없는 삭제 |
| `Request Type`이 PAM, SERVER, Unknown인 정책 포함 그룹 | 별도 검토 필요 유형 |

**출력**: 3개 파일

| 파일명 | 내용 | 포함 컬럼 |
|---|---|---|
| `중복_v1_정리.xlsx` | 전체 분류 결과 (분석 컬럼 모두 포함) | 모든 컬럼 |
| `중복_v1_공지.xlsx` | 공지 대상 (`공지여부 == True`) | Rule Name, No, 작업구분, 신청 정보 (Vsys/Seq/Enable/Action 제외) |
| `중복_v1_삭제.xlsx` | 직접 삭제 대상 (`공지여부 == False`) | 동일 |

### Task 15. 만료셋 예외처리

**목적**: 중복 그룹 중 **전체가 만료된 그룹**이나 **차단 정책 영향을 받는 그룹**을 예외 시트로 분리하여 별도 관리한다.

**입력** (순서 중요):
1. Task 13 출력 정책 파일 (`정책_vf_v2.xlsx`) — 만료여부, 미사용여부 포함
2. Task 14 `_정리.xlsx` — 전체 중복분류 결과
3. Task 14 `_공지.xlsx`
4. Task 14 `_삭제.xlsx`

**전체만료 판별**:
- 동일 `No` 그룹의 **모든 정책**의 `만료여부 == '만료'`이면 → `비고 = 전체만료`

**차단영향위험 판별**:
1. 정책 파일에서 `미사용여부 == '하단최신정책'`인 정책의 `REQUEST_ID` 목록 추출
2. 해당 REQUEST_ID를 포함한 중복 그룹(`No`) 탐색
3. 해당 그룹 내 `삭제` 대상 정책 중 최소 Seq(min_del_seq)와 `유지` 대상 정책 중 최대 Seq(max_keep_seq) 계산
4. `min_del_seq < deny_Seq < max_keep_seq`인 차단 정책이 있으면 → `비고 = 차단영향위험`
   (상단 정책 삭제 시 중간의 차단 정책이 하단 정책에 영향을 미침)

**처리 결과**:
- 예외 대상 `No` 그룹을 `_정리.xlsx`에서 분리하여 `예외` 시트로 이동
- `_공지.xlsx`, `_삭제.xlsx`에서도 해당 `No` 행 제거

**출력**:

| 파일명 | 내용 |
|---|---|
| `중복_v1_정리_v1.xlsx` | 시트1: `중복정책정리` (예외 제거 후), 시트2: `예외` (전체만료/차단영향 그룹) |
| `중복_v1_공지_v1.xlsx` | 예외 그룹 제거된 공지 대상 |
| `중복_v1_삭제_v1.xlsx` | 예외 그룹 제거된 삭제 대상 |
| `duplicate_exceptions.yaml` | 미사용예외 정책 목록 (Task 17 자동 연결용) |

**`duplicate_exceptions.yaml` 형식**:
```yaml
firewall:
  - name: "정책명_A"
    reason: "중복정책삭제_하단노출_임시예외"
    registered_at: "2026-06-14"
    expires_at: "2026-09-12"   # 오늘 + 90일
  - name: "정책명_B"
    ...
```

## 9. Phase 5 — 통보대상 분류 (최종 산출물)

### Task 18. 통보대상 분류

**목적**: 최종 정책 파일을 4가지 유형으로 분류하여 부서 통보용 Excel 파일을 생성한다.

**입력**: Task 17 출력 (`정책_vf_v4.xlsx`)

**필수 컬럼 확인**: `예외`, `중복여부`, `신청이력`, `만료여부`, `미사용여부`

**분류 기준 상세**:

#### ① 기간만료 + 사용 정책 (`_기간만료(공지용).xlsx`)

```
(예외 비어있음 또는 예외 == '신규정책')
AND 중복여부 비어있음
AND 신청이력 != 'Unknown'
AND 만료여부 == '만료'
AND 미사용여부 == '사용'
```

#### ② 기간만료 + 미사용 정책 (`_만료_미사용정책(공지용).xlsx`)

```
(예외 비어있음 또는 예외 == '신규정책')
AND 중복여부 비어있음
AND 신청이력 != 'Unknown'
AND 만료여부 == '만료'
AND 미사용여부 == '미사용'
```

#### ③ 장기미사용 정책 (`_장기미사용정책(공지용).xlsx`)

```
중복여부 비어있음
AND 만료여부 == '미만료'
AND 미사용여부 == '미사용'
AND (
    (신청이력 == 'GROUP' AND 예외 비어있음)
    OR
    (신청이력 == 'GENERAL' AND (예외 비어있음 OR 예외 == '자동연장정책'))
)
```

> GROUP은 자동연장이라도 만료 전 미사용이면 공지 대상.  
> GENERAL은 자동연장 정책은 예외 허용하여 제외함.

#### ④ 이력없는 미사용 정책 (`_이력없는_미사용정책.xlsx`)

```
예외 비어있음
AND 중복여부 비어있음
AND 신청이력 == 'Unknown'
AND 미사용여부 == '미사용'
```

**각 출력 파일의 공통 처리**:
- 설정에 정의된 컬럼만 포함 (`columns.all` 또는 `columns.no_history`)
- 날짜 컬럼을 `YYYY-MM-DD` 형식으로 통일
- 컬럼명 한글 번역 적용 (`translated_columns` 설정 기준)
- 빈 값은 빈 문자열로 치환

## 12. 분류 기준 상세

### 12.1 신청 유형 (Request Type) 판별

정책 이름(`Rule Name`) 또는 설명(`Description`)에 포함된 신청번호 형식으로 판별:

| 첫 글자 | 유형 | 특징 |
|---|---|---|
| `P` | `GROUP` | 그룹 방화벽 정책, 복수 신청자 가능 |
| `F` | `GENERAL` | 일반 방화벽 정책 |
| `S` | `SERVER` | 서버 단위 정책 |
| `M` | `PAM` | PAM (특권접근관리) 연동 정책 |
| (구형 형식) | `OLD` | 레거시 신청 형식 |
| (없음) | `Unknown` | 신청 이력 없음 |

### 12.2 만료 여부 판별

```
REQUEST_END_DATE >= 오늘 날짜 → 미만료
REQUEST_END_DATE <  오늘 날짜 → 만료
날짜 파싱 실패               → 만료 (보수적 처리)
```

### 12.3 미사용 판별 기준

- **기본**: `Unused Days > 90` → `미사용`, `Unused Days <= 90` → `사용`
- **하단최신정책**: Unused Days와 무관, Seq 구조 분석으로 별도 판별
- **중복정책_미사용예외**: 중복 분석 결과 미사용예외로 등록된 정책

### 12.4 중복정책 작업구분 결정

동일 `No` 그룹 내:
- **유지**: `End Date`가 그룹 내 최댓값이고 첫 번째 해당 행 → 최신 신청이 살아있어야 함
- **삭제**: `End Date`가 최댓값이 아닌 모든 행 → 예전 신청은 새 신청에 포섭됨

## 13. 오류 상황 대처

### 13.1 신청정보 파싱 실패 (Request Type = Unknown)

**증상**: 파싱 후에도 `Request Type = Unknown`인 정책이 대량 존재

**원인 및 조치**:
| 원인 | 조치 |
|---|---|
| 신규 신청 형식 도입 | 개발팀에 파싱 패턴(`gsams_3_pattern` 등) 추가 요청 |
| Description 컬럼이 비어있음 | 해당 정책은 `Unknown`으로 유지, 이력없는 미사용 분류에서 별도 관리 |
| 구형 형식 정책 | `gsams_1_*` 패턴 적용 여부 확인 |

### 13.2 Task 15 후 `_정리_v1.xlsx`에 예외 시트가 없음

**증상**: `_정리_v1.xlsx`에 `중복정책정리` 시트만 있고 `예외` 시트가 없음

**원인**: 만료되거나 차단영향 위험이 있는 중복 그룹이 **없는** 경우 정상 동작임.  
예외 시트는 예외 대상이 1건 이상일 때만 생성된다.

> 예외가 있어야 하는데 없다면 → Task 13 정책 파일의 `만료여부`, `미사용여부`, `Action` 컬럼 값을 확인한다.

### 13.3 통보대상 분류 결과가 비어있음 (Task 18)

**증상**: 4개 출력 파일 중 일부 또는 전부가 0행

**확인 사항**:
1. Task 17 출력 파일에 `미사용여부` 컬럼이 존재하는가?
   - 없으면: Task 12 (사용이력 반영) 실패 확인
2. `만료여부` 컬럼 값이 올바른가? (`만료`/`미만료` 두 값만 있어야 함)
3. `예외` 컬럼이 대부분 채워져 있어 분류 대상 자체가 없는 것은 아닌가?
4. `REQUEST_END_DATE` 값이 비어있거나 파싱 실패로 전부 `만료`로 처리되지 않았는가?

### 13.4 GSAMS 매핑 후 날짜가 1900-01-01인 정책 다수 존재

**원인**: Task 8 매핑 시 GSAMS 파일에서 해당 신청번호를 찾지 못한 경우

**조치**:
1. GSAMS 파일 다운로드 범위(기간, 부서) 확인
2. `Request ID` 형식이 GSAMS의 `REQUEST_ID` 형식과 일치하는지 확인
3. 파싱된 `Request ID`에 `v` 접미사가 붙어있으면 자동 제거 로직 확인

### 13.5 자동연장 날짜 업데이트가 안 되는 정책

**확인 사항**:
1. GSAMS 파일에 해당 신청번호의 **연속된** 신청 이력이 있는가?
   - `앞 건 REQUEST_END_DATE == 뒤 건 REQUEST_START_DATE` 조건 충족 여부
2. 제목(`TITLE`)에서 `[TAG]` 제거 후에도 두 건이 동일한가?
3. 작성자(`WRITE_PERSON_ID`)가 두 건 모두 동일한가?
