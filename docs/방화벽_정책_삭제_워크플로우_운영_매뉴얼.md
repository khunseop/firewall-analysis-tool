## 1. 업무 개요

### 1.1 목적

방화벽 **삭제 대상**을 식별하고 정책 신청자 및 결재자에게 사전 공지를 위한 프로세스이다.

삭제 대상은 크게 네 가지 유형이다.

| 삭제 유형 | 판단 기준 |
|---|---|
| **기간만료 + 사용** | 신청 기간이 만료됐으나 여전히 사용이력이 있는 정책 |
| **기간만료 + 미사용** | 신청 기간이 만료됐고 사용이력 없는 정책 |
| **장기미사용** | 기간은 유효하나 90일 이상 사용이력 없는 정책 |
| **이력없는 미사용** | 신청 이력 자체가 없고 사용이력도 없는 정책 |

### 1.2 지원 벤더

- **Palo Alto Networks** (PA 시리즈, HA 구성 포함) — Task 10 처리
- **SECUI MF2 / NGF** — Task 11 처리 (벤더 구분은 시스템이 자동 선택)

## 3. 준비 사항 및 필요 파일

### 3.1 필수 외부 파일

| 파일명 | 획득 방법 | 사용 단계 |
|---|---|---|
| **GSAMS 신청정보 Excel** | GSAMS 담당자에게 요청 | Task 7 |
| **MIS ID 매핑 CSV** | FPMS 담당자에게 요청 | Task 5 |
| **HA Secondary 히트카운트** | Palo Alto HA 구성 시 Secondary 장비에서 별도 추출 | Task 1 (선택) |
| **중복예외 YAML** | 이전 분기 Task 15 산출물 또는 수동 작성 | Task 17 (Task 15 자동 생성 시 불필요) |

### 3.2 MIS ID CSV 형식

```
ruleset_id,mis_id
룰셋아이디1,mis_id_1
룰셋아이디2,mis_id_2
```

> 동일한 `ruleset_id`가 여러 행 있으면 **첫 번째 행만** 사용한다.

### 3.3 작업 환경

- Excel 2016 이상 (`.xlsx` 형식)
- Python 3.10 이상 (자동화 도구 사용 시)
- FAT 시스템 접속 권한

## 4. Phase 0 — 데이터 추출

### Task 0. FAT DB 추출

**목적**: 방화벽 현재 정책 목록과 사용이력(히트카운트)을 한 파일에 담은 기준 데이터를 생성한다.

**처리 절차**:
1. FAT 시스템 → 해당 장비 선택 → "DB 추출" 실행
2. 생성 파일: `YYYY-MM-DD_장비IP_policy.xlsx`

**출력 파일 시트 구성**:

| 시트명 | 내용 |
|---|---|
| `policy` | 정책 목록 (Rule Name, Action, Source, Destination 등) |
| `usage` | 히트카운트 (Rule Name, Hit Count, Last Hit Date, Unused Days) |
| `address` | 주소 객체 |
| `address-group` | 주소 그룹 객체 |
| `service` | 서비스 객체 |
| `service-group` | 서비스 그룹 객체 |

**`policy` 시트 핵심 컬럼**:

| 컬럼명 | 설명 | 예시 |
|---|---|---|
| `Vsys` | 가상 시스템 | `vsys1` |
| `Seq` | 정책 순번 (위에서부터 1, 2, 3…) | `1` |
| `Rule Name` | 정책 이름 | `P2024001234-20241231-hong_gildong` |
| `Enable` | 활성화 여부 | `Y` / `N` |
| `Action` | 허용/차단 | `allow` / `deny` |
| `Source` | 출발지 IP/객체 | `10.0.0.1,10.0.0.2` |
| `Destination` | 목적지 IP/객체 | `192.168.1.0/24` |
| `Service` | 서비스/포트 | `tcp_8080,tcp_443` |
| `Description` | 신청 정보 원문 (파싱 대상) | `P0001234;20240101~20241231;hong;MIS-5678` |

**`usage` 시트 핵심 컬럼**:

| 컬럼명 | 설명 |
|---|---|
| `Rule Name` | 정책 이름 (policy 시트와 조인 키) |
| `Hit Count` | 누적 히트 수 |
| `First Hit Date` | 최초 트래픽 발생일 |
| `Last Hit Date` | 최근 트래픽 발생일 |
| `Unused Days` | 마지막 트래픽 이후 경과 일수 (99999 = 한 번도 사용 안 됨) |

---

### Task 1. HA 히트카운트 병합 (Palo Alto HA 장비 한정, 선택)

**목적**: Active-Passive HA 구성인 경우 Primary와 Secondary 히트카운트를 합산하여 더 정확한 사용이력을 산출한다.

**입력**:
- Task 0 DB 추출 파일 (Primary 히트카운트 포함)
- HA Secondary 장비에서 별도 추출한 히트카운트 파일 (수동 업로드)

**병합 규칙**:

| 컬럼 | 병합 방법 |
|---|---|
| `Hit Count` | Primary + Secondary 합산 |
| `Last Hit Date` | Primary와 Secondary 중 **최신 날짜** 선택 |
| `Unused Days` | Primary와 Secondary 중 **더 적은 값** 선택 (보수적 기준) |
| `미사용여부` | Unused Days > 90일이면 `미사용`, 이하면 `사용` |

**출력**: `Merged_Primary파일명.xlsx`

> HA 구성이 아닌 단일 장비는 Task 1 생략, DB 추출 파일의 `usage` 시트를 직접 사용한다.

## 5. Phase 1 — 정책 파일 전처리

### Task 2. 신청정보 파싱

**목적**: 정책의 `Rule Name` 또는 `Description` 필드에서 신청 정보(신청번호, 신청자, 기간 등)를 추출하여 별도 컬럼으로 분리한다.

**입력**: Task 0 DB 추출 파일 (`policy` 시트)

**파싱 대상 형식**:

| 형식 | 구분 코드 | 신청번호 패턴 | 예시 |
|---|---|---|---|
| GSAMS 3.0 GROUP | `P` | `P` + 7자리 숫자 | `P0001234` |
| GSAMS 3.0 GENERAL | `F` | `F` + 7자리 숫자 | `F0001234` |
| GSAMS 3.0 SERVER | `S` | `S` + 7자리 숫자 | `S0001234` |
| GSAMS 3.0 PAM | `M` | `M` + 7자리 숫자 | `M0001234` |
| GSAMS 1.0 (구형) | `OLD` | 룰이름 또는 Description 내 레거시 패턴 | — |
| 이력없음 | `Unknown` | 위 패턴 없음 | — |

**GSAMS 3.0 `Description` 파싱 구조** (`;` 구분):
```
[그룹1=Ruleset ID];[그룹2=시작일~종료일];[그룹3=신청자];[그룹4=REQUEST_ID];[그룹5=MIS_ID]
예: P0001234;20240101~20241231;hong_gildong;P2024-001234;MIS-5678
```

**추가되는 컬럼**:

| 컬럼명 | 설명 | 비고 |
|---|---|---|
| `Request Type` | 신청 유형 | `GROUP` / `GENERAL` / `SERVER` / `PAM` / `OLD` / `Unknown` |
| `Request ID` | 신청번호 | `P2024-001234` |
| `Ruleset ID` | 룰셋 ID | `P0001234` |
| `MIS ID` | MIS 자산 ID | `MIS-5678` (없으면 빈값) |
| `Request User` | 신청자 사번/ID | `hong_gildong` |
| `Start Date` | 신청 시작일 | `2024-01-01` |
| `End Date` | 신청 종료일 | `2024-12-31` |

**출력**: `정책파일_v1.xlsx`

### Task 3. 중복정책 분석 (FAT 자동)

**목적**: 동일한 출발지·목적지·서비스 조합을 가진 중복 정책을 탐지하고 그룹 번호(`No`)를 부여한다.

**처리**: FAT 시스템 내부 분석 엔진이 자동 처리 (별도 조작 불필요)

**출력 컬럼**:

| 컬럼명 | 설명 |
|---|---|
| `No` | 중복 그룹 번호 (같은 No끼리 중복 관계) |
| `Type` | `Upper` (상단, 실제 적용되는 정책) / `Lower` (하단, 가려진 정책) |
| 기존 정책 컬럼 | 그대로 유지 |

### Task 4. 중복결과 신청정보 파싱

**목적**: Task 3 중복분석 결과 파일에도 신청정보를 파싱하여 추가한다.

**입력**: Task 3 출력 파일  
**처리**: Task 2와 동일한 파싱 로직 적용  
**출력**: `중복결과_v1.xlsx`

### Task 5. MIS ID 매핑

**목적**: FPMS의 MIS ID를 `Ruleset ID` 기준으로 정책 파일에 채워넣는다.

**입력**:
- Task 2 출력 (`정책_v1.xlsx`)
- MIS ID 매핑 CSV (수동 업로드)

**매핑 규칙**:
- `Ruleset ID`를 키로 CSV에서 `mis_id`를 조회
- 정책 파일의 `MIS ID` 컬럼이 비어있는 행에만 채움 (기존 값 보존)
- CSV에 중복 `ruleset_id`가 있으면 **첫 번째 항목만** 사용

**출력**: `정책_v2.xlsx`

---

### Task 6. 신청번호 추출

**목적**: 이후 GSAMS 조회에 사용할 신청번호 목록을 `Request Type`별 시트로 정리한다.

**입력**: Task 5 출력 (`정책_v2.xlsx`)

**처리 로직**:
1. `Request Type != 'Unknown'`인 행만 선택
2. 고유한 `Request Type` 목록 추출 (최대 5종)
3. 각 타입별로 `Request ID` 고유값 목록을 별도 시트로 저장

**출력**: `request_id_정책파일명.xlsx`

| 시트명 | 내용 |
|---|---|
| `GROUP` | P로 시작하는 그룹 신청번호 목록 |
| `GENERAL` | F로 시작하는 일반 신청번호 목록 |
| `SERVER` | S로 시작하는 서버 신청번호 목록 |
| `PAM` | M으로 시작하는 PAM 신청번호 목록 |
| `OLD` | 구형 형식 신청번호 목록 (존재 시) |

## 6. Phase 2 — 신청정보 매핑

### Task 7. GSAMS 신청정보 취합

**목적**: GSAMS 신청 정보파일을 표준 포맷으로 변환·통합한다.

**입력**: GSAMS 원본 Excel (수동 업로드, 멀티시트 가능)

**처리**:
1. 각 시트를 순회하여 컬럼명을 내부 표준 컬럼명으로 매핑
2. 이메일 주소 자동 생성: `WRITE_PERSON_ID@[도메인]`, `REQUESTER_ID@[도메인]`
3. `REQUEST_START_DATE`, `REQUEST_END_DATE`: YYYYMMDD → YYYY-MM-DD 변환
4. 전체 시트를 하나로 합치고 `REQUEST_END_DATE` 내림차순 정렬

**출력**: `Conv_GSAMS파일명.xlsx`

**핵심 컬럼**:

| 컬럼명 | 설명 |
|---|---|
| `REQUEST_ID` | 신청번호 |
| `TITLE` | 신청 제목 (자동연장 판별에 사용) |
| `REQUEST_START_DATE` | 신청 시작일 (YYYY-MM-DD) |
| `REQUEST_END_DATE` | 신청 종료일 (YYYY-MM-DD) |
| `REQUEST_STATUS` | `99` = 자동연장 신청 표시 |
| `WRITE_PERSON_ID` | 기안자 ID |
| `REQUESTER_ID` | 신청자 ID |
| `REQUESTER_EMAIL` | 신청자 이메일 |
| `MIS_ID` | 자산 ID |

---

### Task 8. 신청정보 정책파일 매핑

**목적**: GSAMS 취합 파일의 상세 신청 정보를 정책 파일의 각 정책에 매핑한다.

**입력**:
- Task 5 출력 (`정책_v2.xlsx`)
- Task 7 출력 (`Conv_*.xlsx`)

**매핑 우선순위** (GROUP 유형, 순서대로 시도):

| 순위 | 매핑 기준 |
|---|---|
| 1순위 | `REQUEST_ID` 일치 AND `MIS_ID` 일치 |
| 2순위 | `REQUEST_ID` 일치 AND `REQUEST_END_DATE` 일치 AND `WRITE_PERSON_ID` 일치 |
| 3순위 | `REQUEST_ID` 일치 AND `REQUEST_END_DATE` 일치 AND `REQUESTER_ID` 일치 |

**매핑 우선순위** (GENERAL / SERVER / PAM / OLD / Unknown 유형):
- `REQUEST_ID` 일치만으로 매핑

**자동연장 표시 규칙**:
- 매핑된 GSAMS 레코드의 `REQUEST_STATUS == 99`이면 → `REQUEST_STATUS` 컬럼에 `99` 저장

**매핑 실패 시 처리**:
- Task 2에서 파싱된 `Request ID`, `Start Date`, `End Date`, `Request User` 값을 그대로 사용

### Task 9. 자동연장 날짜 업데이트

**목적**: 자동연장이 반복된 정책의 `REQUEST_END_DATE`를 가장 최신 연장일로 갱신한다.

**자동연장 체인 판별 로직** (GSAMS 파일 내 self-join):

```
전제조건 (세 가지 모두 만족해야 체인으로 판정):
  1. 앞 건의 REQUEST_END_DATE == 뒤 건의 REQUEST_START_DATE

```

> **괄호 머리말 제거 예시**: `[위험] 방화벽 정책 신청` → `방화벽 정책 신청`

| 예외 값 | 분류 조건 |
|---|---|

### Task 13. 하단 최신정책 검증

**목적**: 동일 신청번호(`REQUEST_ID`) 그룹 내에서 **최신 신청일의 정책이 더 하단(Seq가 큰 쪽)에 위치**한 경우를 탐지하고 `미사용여부 = '하단최신정책'`으로 표시한다.

**입력**: Task 12 출력 (`정책_vf_v1.xlsx`)

**분석 대상**: `신청이력 != 'Unknown'`인 정책만 분석

**탐지 로직**:
```
각 REQUEST_ID 그룹에 대해:
  1. REQUEST_START_DATE의 최댓값 탐색
     → 해당 날짜의 정책들의 Seq 목록 = latest_seqs
  2. 그룹 내 가장 작은 Seq = lowest_seq
  3. lowest_seq가 latest_seqs 목록에 없으면:
     → 최신 신청 정책이 하단에 위치하는 상황
     → latest_seqs에 해당하는 정책들의 미사용여부 = '하단최신정책'
```

**실무 의미**:  
동일 신청번호의 이전 정책(갱신 전)이 상단에, 최신 정책(갱신 후)이 하단에 있으면, 실제 트래픽은 상단 정책에 의해 처리되므로 하단의 최신 정책이 사용 중처럼 보여도 실제로는 가려진 상태다.

**출력**: `정책_vf_v2.xlsx` (멀티시트)

| 시트명 | 내용 |
|---|---|
| `usage` | 메인 정책 데이터 (`미사용여부` 업데이트 포함) |
| `검증` | 탐지 결과 (`REQUEST_ID`, `lowest_seq`, `latest_seq` 목록) |

### Task 14. 중복정책 분류

**목적**: Task 3에서 탐지된 중복 그룹을 분류하여 삭제 대상과 유지 대상을 구분하고 공지용/삭제용 파일을 생성한다.

**입력**:

2. Task 13 출력 (`정책_vf_v2.xlsx`) — `미사용여부`, `REQUEST_STATUS`, `REQUESTER_EMAIL` 포함 정책

**분류 컬럼 계산** (중복분석 파일에 추가):

| 컬럼명 | 계산 방법 |
|---|---|
| `Requester Email` | 정책 파일의 `Rule Name → REQUESTER_EMAIL` 매핑으로 채움 |
| `자동연장` | `Request ID`가 정책 파일의 `REQUEST_STATUS==99` 레코드의 `REQUEST_ID`에 있으면 `True` |
| `늦은종료일` | 동일 `No` 그룹 내에서 `End Date`가 최댓값이고 **첫 번째** 해당 행 = `True` |
| `신청자검증` | 동일 `No` 그룹 내 `Request User`가 모두 동일하면 `True` |
| `날짜검증` | `Type == 'Upper'` AND `늦은종료일 == True`인 `No`에 속하면 `True` |
| `작업구분` | `늦은종료일 == True` → `유지`, `False` → `삭제` |
| `공지여부` | `신청자검증 == False` (신청자가 다른 경우) → `True` |
| `미사용예외` | `날짜검증 == False` AND `늦은종료일 == True` → `True` |

**`날짜검증` 상세**:
- 동일 No 그룹 내에 `Type == 'Upper'` AND `늦은종료일 == True`인 정책이 있으면 → 그룹 전체 `날짜검증 = True`
- 이 조건은 "실제 적용되는 상단 정책이 가장 최신 신청"임을 의미 (정상 구성)
- 그렇지 않으면 `날짜검증 = False` → `미사용예외 = True`

**예외 그룹 제거 규칙** (분류 후 최종 결과에서 완전히 제외):

| 제거 조건 | 이유 |
|---|---|
| 자동연장 GROUP이면서 동일 No 내 다른 REQUEST_ID가 2개 이상이고 삭제 대상이 있는 경우 | 연장 체인 구조, 함부로 삭제 불가 |
| 자동연장이면서 GROUP 아닌 유형이 섞여있고 삭제 대상이 있는 그룹 | 자동연장 정책 보호 |
| `No` 그룹 내 `작업구분 == '유지'`인 정책이 하나도 없는 경우 | 유지 대상 없으면 의미없는 분류 |
| `Request Type`이 `PAM`, `SERVER`, `Unknown`인 정책 포함 그룹 | 별도 검토 필요 유형 |

**공지/삭제 파일에서 제거되는 컬럼**:
- `Request Type`, `Ruleset ID`, `MIS ID`, `Start Date`, `End Date`
- `늦은종료일`, `신청자검증`, `날짜검증`, `공지여부`, `미사용예외`, `자동연장`
- `Vsys`, `Seq`, `Enable`, `Action`

**출력**: 3개 파일 (알파벳순 정렬 기준 슬롯 할당)

| 파일명 | 내용 | 슬롯 |
|---|---|---|
| `중복_v1_공지.xlsx` | 공지 대상 (`공지여부 == True`) | output_0 |
| `중복_v1_삭제.xlsx` | 직접 삭제 대상 (`공지여부 == False`) | output_1 |
| `중복_v1_정리.xlsx` | 전체 분류 결과 (분석 컬럼 모두 포함) | output_2 |

> 슬롯 순서는 한글 유니코드 정렬 결과: 공(U+ACF5) < 삭(U+C0AD) < 정(U+C815)

### Task 15. 만료셋 예외처리

**목적**: 중복 그룹 중 **전체가 만료된 그룹** 또는 **차단 정책 영향을 받는 그룹**을 예외 시트로 분리하고, 미사용예외 정책은 YAML로 저장한다.

**입력 (순서 중요)**:
1. Task 13 출력 정책 파일 (`정책_vf_v2.xlsx`) — `만료여부`, `미사용여부`, `Action` 포함
2. Task 14 `_정리.xlsx` (output_2) — 전체 중복분류 결과
3. Task 14 `_공지.xlsx` (output_0)
4. Task 14 `_삭제.xlsx` (output_1)

**전체만료 판별**:
```
동일 No 그룹의 모든 정책에서 '만료여부 == 만료' → 비고 = '전체만료'
```

**차단영향위험 판별**:
```
1. 정책 파일에서 미사용여부 == '하단최신정책'인 정책의 REQUEST_ID 추출
2. 해당 REQUEST_ID를 포함한 중복 그룹(No) 탐색
3. 해당 그룹에서:
   - 삭제 대상 정책(작업구분 == '삭제')의 최솟값 Seq = min_del_seq
   - 유지 대상 정책(작업구분 == '유지')의 최댓값 Seq = max_keep_seq
4. 정책 파일의 차단정책(Action == 'deny')의 Seq 목록 중
   min_del_seq < deny_Seq < max_keep_seq 인 것이 있으면:
   → 비고 = '차단영향위험'
   (상단 정책 삭제 시 그 아래 차단 정책이 하단 유지 정책에 영향을 미칠 위험)
```

**처리 결과**:
- 예외 대상 No 그룹을 `_정리.xlsx`에서 분리하여 `예외` 시트로 이동
- `_공지.xlsx`, `_삭제.xlsx`에서도 해당 No 행 제거
- `예외` 시트의 `비고` 컬럼에 `전체만료` 또는 `차단영향위험` 표시

**출력**:

| 파일명 | 내용 | 슬롯 |
|---|---|---|
| `중복_v1_공지_v1.xlsx` | 예외 그룹 제거된 공지 대상 | output_0 |
| `중복_v1_삭제_v1.xlsx` | 예외 그룹 제거된 삭제 대상 | output_1 |
| `중복_v1_정리_v1.xlsx` | 시트1: `중복정책정리`, 시트2: `예외` | output_2 |
| `duplicate_exceptions.yaml` | 미사용예외 정책 YAML (Task 17 자동 연결) | output_3 |

**`duplicate_exceptions.yaml` 생성 조건**:
- `_정리.xlsx`의 `미사용예외 == True`인 정책이 1건 이상인 경우에만 생성
- 유효기간 = `오늘 + 90일`

**`duplicate_exceptions.yaml` 형식**:
```yaml
firewall:         # 장비명 (시스템 설정의 firewall_name 또는 기본값 'firewall')
  - name: "정책명_A"
    reason: "중복정책삭제_하단노출_임시예외"
    registered_at: "2026-06-14"     # 오늘 날짜
    expires_at: "2026-09-12"        # 오늘 + 90일
  - name: "정책명_B"
    ...
```

> **예외 시트가 생성되지 않는 경우**: 전체만료 그룹도 없고 차단영향위험 그룹도 없으면 예외 시트가 없는 것이 **정상 동작**이다.

### Task 16. 중복정책 상태 업데이트

**목적**: 중복분류 결과의 `작업구분`(유지/삭제)을 메인 정책 파일에 `중복여부` 컬럼으로 반영한다.

**입력**:
1. Task 13 출력 정책 파일 (`정책_vf_v2.xlsx`) — `미사용여부` 포함
2. Task 14 출력 중복분류 결과 — `Rule Name`, `작업구분` 컬럼 필요

**처리 로직**:
```
1. 중복분류 파일의 Rule Name → 작업구분 매핑 생성
2. 정책 파일에 '중복여부' 컬럼 추가
3. '중복여부' 컬럼을 파일의 두 번째 컬럼 위치로 이동
```

## 9. Phase 5 — 통보대상 분류 (최종 산출물)

### Task 18. 통보대상 분류

**목적**: 최종 정책 파일을 4가지 삭제 유형으로 분류하여 부서 통보용 Excel 파일을 생성한다.

**입력**: Task 17 출력 (`정책_vf_v4.xlsx`)

**분류 기준 상세**:

#### ① 기간만료 + 사용 정책 → `_기간만료(공지용).xlsx`

```
(예외 가 NaN 이거나 예외 == '신규정책')
AND 중복여부 가 NaN
AND 신청이력 != 'Unknown'
AND 만료여부 == '만료'
AND 미사용여부 == '사용'
```

#### ② 기간만료 + 미사용 정책 → `_만료_미사용정책(공지용).xlsx`

```
(예외 가 NaN 이거나 예외 == '신규정책')
AND 중복여부 가 NaN
AND 신청이력 != 'Unknown'
AND 만료여부 == '만료'
AND 미사용여부 == '미사용'
```

#### ③ 장기미사용 정책 → `_장기미사용정책(공지용).xlsx`

```
중복여부 가 NaN
AND 만료여부 == '미만료'
AND 미사용여부 == '미사용'
AND (
    (신청이력 == 'GROUP'   AND 예외 가 NaN)
    OR
    (신청이력 == 'GENERAL' AND (예외 가 NaN OR 예외 == '자동연장정책'))
)
```

> GROUP은 자동연장이라도 만료 전 미사용이면 공지 대상.  
> GENERAL은 자동연장 정책인 경우 예외 처리하여 공지 제외.

#### ④ 이력없는 미사용 정책 → `_이력없는_미사용정책.xlsx`

```
예외 가 NaN
AND 중복여부 가 NaN
AND 신청이력 == 'Unknown'
AND 미사용여부 == '미사용'
```

**각 출력 파일의 공통 처리**:
- 설정 `columns.all`에 정의된 컬럼만 포함 (①②③), `columns.no_history`는 ④에 적용
- 날짜 컬럼(`date_columns` 설정)을 `YYYY-MM-DD` 형식으로 통일
- 컬럼명 한글 번역 적용 (`translated_columns` 설정 기준)
- `NaN` 및 문자열 `'nan'`을 빈 문자열로 치환

## 12. 분류 기준 상세

### 12.1 신청 유형 (Request Type) 판별

| 첫 글자 | 유형 | 특징 |
|---|---|---|
| `P` | `GROUP` | 그룹 방화벽 정책, 복수 신청자 가능, MIS ID 매핑 대상 |
| `F` | `GENERAL` | 일반 방화벽 정책, 자동연장 예외 조건 있음 |
| `S` | `SERVER` | 서버 단위 정책, 중복분류에서 제외 |
| `M` | `PAM` | PAM 연동 정책, 중복분류에서 제외 |
| (구형 형식) | `OLD` | 레거시 신청 형식 |
| (없음) | `Unknown` | 신청 이력 없음, 이력없는 미사용 분류 대상 |

### 12.2 만료 여부 판별

```
REQUEST_END_DATE >= 오늘 날짜 → 미만료
REQUEST_END_DATE <  오늘 날짜 → 만료
날짜 파싱 실패               → 만료 (보수적 처리)
```

### 12.3 미사용 판별 기준

| `미사용여부` 값 | 판별 방법 | 적용 시점 |
|---|---|---|
| `사용` | `Unused Days <= 90` | Task 12 |
| `미사용` | `Unused Days > 90` | Task 12 |
| `하단최신정책` | Seq 구조 분석 (Task 13 별도 판별) | Task 13에서 덮어씀 |
| `중복정책_미사용예외` | YAML 기반 (Task 17) | Task 17에서 덮어씀 |

### 12.4 중복정책 작업구분 결정

동일 `No` 그룹 내:
- **유지**: `End Date`가 그룹 최댓값이고 그 중 첫 번째 해당 행
- **삭제**: `End Date`가 최댓값이 아닌 모든 행

### 12.5 YAML 예외 유효성 기준

```
유효: registered_at < 오늘 AND expires_at >= 오늘
무효: expires_at < 오늘 (만료된 예외)
무효: registered_at >= 오늘 (당일 또는 미래 등록, 즉시 유효하지 않음)
```

## 13. 오류 상황 대처

### 13.1 신청정보 파싱 실패 (Request Type = Unknown) 다수

| 원인 | 조치 |
|---|---|
| 신규 신청 형식 도입 | 개발팀에 파싱 패턴 추가 요청 |
| Description 컬럼이 비어있음 | `Unknown`으로 유지, 이력없는 미사용 분류에서 별도 관리 |
| 구형 형식 정책 | 구형 패턴 적용 여부 확인 |

### 13.2 Task 15 후 `_정리_v1.xlsx`에 예외 시트가 없음

전체만료 그룹도 없고 차단영향위험 그룹도 없으면 **정상 동작**이다.  
예외가 있어야 하는데 없다면:
- Task 13 정책 파일의 `만료여부` 컬럼 확인 (전체만료 판별용)
- Task 13 정책 파일의 `미사용여부 == '하단최신정책'` 존재 여부 확인 (차단영향위험 판별용)
- Task 13 정책 파일의 `Action` 컬럼 확인 (deny 정책 Seq 비교용)

### 13.3 통보대상 분류 결과가 비어있음 (Task 18)

1. Task 17 출력에 `미사용여부` 컬럼 존재 여부 확인 (없으면 Task 12 실패)
2. `만료여부` 컬럼 값이 `만료`/`미만료` 두 값만 있는지 확인
3. `예외` 컬럼이 대부분 채워져 있어 분류 대상 자체가 없는 것은 아닌지 확인
4. `REQUEST_END_DATE` 파싱 실패로 전부 `만료` 처리되지 않았는지 확인

### 13.4 GSAMS 매핑 후 날짜가 1900-01-01인 정책 다수

해당 신청번호를 GSAMS에서 찾지 못한 경우다.

1. GSAMS 파일 다운로드 범위(기간, 부서) 확인
2. `Request ID` 형식이 GSAMS의 `REQUEST_ID` 형식과 일치하는지 확인
3. 파싱된 `Request ID`에 버전 접미사(`_v1` 등)가 붙어있으면 제거 로직 확인

### 13.5 자동연장 날짜 업데이트가 안 되는 정책 (Task 9)

1. GSAMS 파일에 해당 신청번호의 **연속된** 신청 이력이 있는가?
   - `앞 건 REQUEST_END_DATE == 뒤 건 REQUEST_START_DATE` 조건 충족 여부
2. 제목(`TITLE`)에서 `[TAG]` 제거 후에도 두 건이 동일한가?
3. 작성자(`WRITE_PERSON_ID`)가 두 건 모두 동일한가?

### 13.6 중복여부가 채워지지 않는 정책 있음 (Task 16)

중복분류 결과에 해당 Rule Name이 없으면 `중복여부`가 NaN으로 남는다.  
Task 14에서 예외 그룹 제거 과정에서 필터링된 경우이므로 **정상 동작**이다.

### 13.7 YAML 예외가 반영되지 않음 (Task 17)

1. `registered_at`이 오늘 날짜인 경우: `registered_at < 오늘` 조건에 의해 **당일 등록은 무효**
2. `expires_at`이 오늘 이전인 경우: 만료된 예외는 적용되지 않음
3. `name` 값이 정책 파일의 `Rule Name`과 정확히 일치하는지 확인 (공백, 대소문자 포함)
